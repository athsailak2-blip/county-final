## 1. Origin and purpose

Phase 5 was the first source-level recon for a non-trivial source тАФ
`publicsearch_clerk_recordings`, the Bexar County Clerk recordings on the PublicSearch
portal. It exercised the framework end-to-end on paper: a probe of the live portal, a
124-code doc-type catalog, a doc-type classification with a 20-code daily-refresh list, a
locked v2 scraper spec, and a locked v1 translator spec.

That end-to-end pass surfaced real gaps in three framework areas:

- **Canonical vocabulary** тАФ the translator needs canonical doc types and canonical
  record-field names that do not yet exist in the framework's registries.
- **Contract specification** тАФ the translator output contract is real in code but not
  documented in `MASTER_PROMPT.md`; some conventions (flag attachment, unknown-doc-type
  handling) are unspecified.
- **Framework tooling** тАФ a missing standalone `validate()` API, and a regression
  scanner that flags legitimate non-county content.

This file catalogs those gaps with enough precision that a future v5.1.2-beta-final
cleanup session can execute against it without re-doing the recon.

## 3. Missing canonical doc types (CANONICAL, REQUIRED тАФ item A)

**Source:** `publicsearch_clerk_recordings_translator_spec.md` ┬з4 mapping table, the GAP
entries. Cross-checked: each proposed name below was confirmed **absent** from the
74-entry `canonical_types` block in `knowledge_base/domain/canonical_doc_types.json`.

The translator spec mapped 20 daily-refresh PublicSearch codes to canonical doc types: 5
matched an existing canonical, **15 did not** and were flagged GAP. The 15 GAP canonicals
must be added to `canonical_doc_types.json` in v5.1.2-beta-final. They are not invented
here тАФ every name is exactly what the translator spec ┬з4 proposed.

Each entry: proposed canonical тАФ maps from тАФ justification тАФ specific-event vs umbrella тАФ
universality.

- **`PROBATE_RECORDING`** тАФ from `PROBATE` тАФ a generic probate filing recorded against
  property; no probate canonical exists today тАФ **umbrella** тАФ universal (every county
  records probate).
- **`PROBATE_LETTERS`** тАФ from `LETTERS` тАФ letters testamentary / of administration;
  `LETTERS_TESTAMENTARY` and `LETTERS_OF_ADMINISTRATION` exist individually but the
  PublicSearch `LETTERS` code does not distinguish them, so an umbrella is needed тАФ
  **umbrella** (over two existing specific canonicals) тАФ universal.
- **`WILL_RECORDING`** тАФ from `WILL` (label `WILL & TESTAMENT`) тАФ a recorded will; no
  `WILL` canonical exists (`MUNIMENT_OF_TITLE` is adjacent but distinct) тАФ
  **specific-event** тАФ universal.
- **`COURT_DECREE`** тАФ from `DECREE` тАФ a generic court decree; `FINAL_DECREE_OF_DIVORCE`
  exists but `DECREE` is a broad umbrella тАФ **umbrella** тАФ universal.
- **`GENERIC_AFFIDAVIT`** тАФ from `AFFIDAV` тАФ the affidavit umbrella;
  `AFFIDAVIT_OF_HEIRSHIP` exists but most affidavits are not heirship, and heirship
  detection is Level 3 тАФ **umbrella** тАФ universal.
- **`CHILD_SUPPORT_LIEN`** тАФ from `CSUP LN` тАФ a child-support lien; no such canonical
  exists тАФ **specific-event** тАФ universal (a recognized lien type nationwide).
- **`COURT_JUDGMENT`** тАФ from `JUDG` тАФ a generic judgment; `JUDGMENT_LIEN` and
  `VACATED_JUDGMENT` exist but `JUDG` is the umbrella and not every judgment is a
  property lien тАФ **umbrella** тАФ universal.
- **`STATE_JUDGMENT`** тАФ from `SJ` (label `State-Judgment`) тАФ a state-court judgment
  record; the `SJ` code is Bexar-specific naming but the underlying concept is not тАФ
  **specific-event** тАФ universal in concept (the Bexar code label is local; the canonical
  is generic).
- **`GENERIC_LIEN`** тАФ from `LIEN` тАФ the broad lien umbrella; many specific lien
  canonicals exist (`CONSTRUCTION_LIEN`, `HOA_LIEN`, `MUNICIPAL_LIEN`, `JUDGMENT_LIEN`,

- **`LANDLORD_LIEN`** тАФ from `LNLD LN` тАФ a landlord lien; no such canonical exists тАФ
  **specific-event** тАФ universal.
- **`POWER_OF_ATTORNEY`** тАФ from `PA` тАФ a recorded power of attorney; no such canonical

- **`GENERIC_MEMORANDUM`** тАФ from `MEMO` тАФ the memorandum umbrella (memo of lease / of

- **`GENERIC_NOTICE`** тАФ from `NOTICE` тАФ the notice umbrella; several specific notice

## 4. Canonical record fields vocabulary (CANONICAL, REQUIRED тАФ item B)

**Source:** the Bexar migration backlog plus `publicsearch_clerk_recordings_translator_
spec.md` ┬з6 (the `field_map` TBDs).

**Problem.** Canonical *record-field* names тАФ `address`, `doc_number`, `owner_name`,
`recording_year`, `recording_month`, `city`, `zip`, and so on тАФ are used informally
throughout the framework. `MASTER_PROMPT.md ┬з4.32` itself flags this: it says a
"canonical-field-name registry (`scaffold/data/canonical_record_fields.json` or
similar)" is "on the v5.1.2-beta-final backlog", and instructs scrapers to match field
names "used in existing v5.1.2-beta-r2+ canonical translators" in the meantime. Today the
only way to learn a canonical field name is to read an existing translator's docstring.
Translators and source configs therefore guess at canonical names.

**Proposed fix.** Create `scaffold/data/canonical_record_fields.json` тАФ a sibling of
`canonical_doc_types.json` тАФ cataloguing every canonical record-field name with: the
canonical name; type (string / integer / date / boolean); required vs. optional; a brief
description; and examples. Once it exists, translators and validators treat it as the
single source of truth, exactly as `canonical_doc_types.json` is the source of truth for
doc types.

**Specific naming questions this file must resolve** (all surfaced as TBDs in translator
spec ┬з6 / ┬з15 open question 3):

- **`recording_date`** тАФ the existing `foreclosure_notices` translator reads split
  integer fields `recording_year` + `recording_month` (no day). The PublicSearch scraper
  emits a single `recorded_date` as `YYYY-MM-DD`. The registry must decide: a single
  canonical `recording_date` (full date), OR keep the split `recording_year` /
  `recording_month` / `recording_day` model. A clerk-recordings source has a precise day;
  the foreclosure-notices source historically did not.
- **`grantor` / `grantee`** тАФ `foreclosure_notices` uses a single `owner_name`. Clerk
  recordings have two parties. The registry must decide: two parallel canonical fields
  (`grantor`, `grantee`), OR a single role-tagged structure (`owner_parties_raw`).
  `canonical_doc_types.json` already has a 16-entry `party_roles` list, which hints at a
  role-tagged model.
- **`parcel_grid_identifiers`** тАФ the scraper emits one raw concatenated string (Lot /
  Block / NCB / County Block). The registry must decide: a single canonical raw-string
  field (e.g. `parcel_identifier_raw`), OR split canonical sub-fields (`lot`, `block`,
  `ncb`, `county_block`). The translator spec recommends single-string for v1.
- **`source_url` / `source_fetched_at` / `parser_confidence`** тАФ these are already
  canonical top-level metadata per `┬з4.32`. For completeness, the registry should still
  list them (marked as framework-metadata rather than `raw_payload` fields), so one
  document describes the entire canonical vocabulary.

This item is REQUIRED because it blocks authoring the Bexar `field_map` (and every future
source's `field_map`) тАФ you cannot write a `field_map` whose keys are canonical names
until the canonical names are defined.

## 5. Translator output contract clarification (CONTRACT, DOCUMENTATION тАФ item C)

**Source:** `publicsearch_clerk_recordings_translator_spec.md` ┬з3.2 and ┬з15 open
question 1.

**Finding.** The translator spec assumed (per its step-5 instructions) a translator
output of flat canonical records with `canonical_record_id` / `canonical_payload` /
`field_map_applied` / `record_emitted_at`. Reading the actual framework тАФ
`scaffold/pipeline/translators/__init__.py` and the three built-in translators
`foreclosure_notices.py`, `parcel_master.py`, `csv_static_list.py` тАФ every registered
translator instead returns a **three-tuple**:

    translator_fn(raw_records, county_config, source_config)
        -> (signals, parcels, per_signal_meta_by_url)

typed `tuple[list[dict], list[dict], dict[str, dict]]`. This is the real translator
output contract. `MASTER_PROMPT.md` documents the *input* side (┬з4.32, wrapped
`raw_payload`) but does not explicitly document this *output* side тАФ the three-tuple
shape is currently only implicit in the translator implementations.

(Note: the step-5 instructions referenced filenames `arcgis_foreclosure_notices.py` /
`arcgis_parcel_master.py`; the actual files dropped the `arcgis_` prefix in
v5.1.2-beta-r2 and are `foreclosure_notices.py` / `parcel_master.py`. The contract is the
same regardless of filename.)

**Proposed fix.** Add a `MASTER_PROMPT.md` subsection тАФ e.g. **┬з4.33** or appended to
┬з4.32 тАФ that explicitly documents:

- the three-tuple translator output contract and the `(raw_records, county_config,
  source_config)` call signature;
- what each tuple element is тАФ `signals` (one per translated record), `parcels`
  (deduplicated placeholder parcels), `per_signal_meta_by_url` (keyed by `source_url`) тАФ
  including the expected fields of each, and when each is empty;
- concrete examples drawn from `foreclosure_notices.py` or `parcel_master.py`;
- a cross-link: ┬з4.32 (wrapped `raw_payload` input) тЖТ ┬з4.33 (translator output).

This item is **DOCUMENTATION only** тАФ the contract already exists and is exercised by
`test_translator_registry.py`. The addition makes it explicit for future translator
authors so they do not have to reverse-engineer it from existing code (as the Phase 5
translator spec had to). No translator code or test changes are needed.

## 6. Level 2 flag attachment in the three-tuple (CONTRACT, REQUIRED тАФ item D)

**Source:** `publicsearch_clerk_recordings_translator_spec.md` ┬з5 and ┬з15 open
question 2.

**Question.** The translator emits two Level 2 flags тАФ `lifecycle_suppression_flag` and
`dedup_against_foreclosure_notices_map`. The translator output is the three-tuple from
┬з5. Neither flag is part of the existing `foreclosure_notices.py` signal shape, so the
attachment point is unspecified. Three candidates:

- **Inside each `signal` record** тАФ a per-signal boolean field. (The translator spec
  provisionally chose this.)
- **Inside `per_signal_meta_by_url`** тАФ e.g. as entries in the existing
  `preset_review_flags` list (flag strings such as `"lifecycle_suppression"` /
  `"fc_dedup_required"`).
- **As a separate top-level element of the tuple** тАФ would expand the contract from a
  three-tuple to a four-tuple, affecting every translator and `test_translator_registry`.

**Proposed fix.** v5.1.2-beta-final must pick one and document it as part of the ┬з4.33
contract addition (item C). **Recommendation (not locked here):** attach as per-signal
fields on the `signal` dict тАФ a lifecycle/dedup flag is a signal-level attribute, not
URL-level metadata, and adding a tuple element would be a disruptive contract change. The
final decision is the cleanup session's to make.

This item is REQUIRED because the translator code cannot be written until the attachment
point is fixed тАФ the flags are core Level 2 output.

## 7. unknown_doc_type_handling default (CONFIG, RECOMMENDED тАФ item E)

**Source:** `publicsearch_clerk_recordings_translator_spec.md` ┬з11 and ┬з15 open
question 4.

**Question.** When a translator encounters a `doc_type_code` it cannot resolve to a
canonical doc type, what is the default behavior? Three options:

- **`skip_with_warning`** тАФ drop the record, log a warning, continue.
- **`include_with_canonical_unknown`** тАФ emit the record with
  `doc_type = UNKNOWN_DOC_TYPE`, log a warning, continue.
- **`halt_run`** тАФ treat it as an upstream bug and halt the translator.

**Proposed fix.** v5.1.2-beta-final should pick a single framework default and document
it as a translator-registry convention (alongside the ┬з4.33 contract). Operators can
override per-source via `source_config`. **Recommendation (not locked here):**
`skip_with_warning` as the framework default тАФ defensive (a bad code never corrupts
downstream output) but visible (the warning surfaces the problem). The decision is queued
for the cleanup session, not made here.

## 8. validate(target, schema_path) helper API (TOOLING, REQUIRED тАФ item F)

**Source:** the Bexar migration backlog тАФ
`runs/bexar_tx/backlog/v5.1.2-beta-framework-patches.md` item 3 and
`v5.1.2-beta-migration-ambiguities.md`.

**Problem (evidence-confirmed).** `scaffold/ops/write_county_config.py` exposes only
`write_county_config(config_dict, target_path, schema_path=None, overwrite=False)` тАФ a
write-and-validate-in-one-call function. There is no standalone, read-only validation
function. The migration backlog records that documentation and tooling reference a
`validate()` that does not exist:

    python3 -c "from scaffold.ops.write_county_config import validate; \
                print(validate('config/counties/bexar_tx.json'))"
    # -> ImportError: cannot import name 'validate'

Tests and external callers that want to validate a config (or any target JSON) against a
schema without writing a file have no clean entry point.

**Proposed fix (per the migration backlog).** Add a standalone, read-only `validate()`
in `scaffold/ops/write_county_config.py` (or a new `scaffold/ops/validators.py`):

- accepts a target file path or a dict, plus a schema path;
- does NOT write;
- returns a structured result тАФ the migration backlog proposes a `ValidateResult`
  dataclass with `status` (`OK` / `JSON_INVALID` / `SCHEMA_INVALID`) and
  `schema_validation` (`VALIDATED` / `SCHEMA_VALIDATION_SKIPPED` / `SCHEMA_INVALID` /
  `SCHEMA_FILE_MISSING`);
- is reused by `test_translator_registry`, `test_golden_path`, source-config CI, and the
  `test_write_county_config.py` cases the migration backlog already enumerates;
- adds `validate` and `ValidateResult` to the module `__all__`;
- is documented in `MASTER_PROMPT.md` as a framework helper.

**Justification.** Removes duplicated ad-hoc validation logic across tests and tooling
and gives a single read-only validation entry point. REQUIRED because pending framework
tests and CI assume it exists.

## 9. Regression scanner exemptions (TOOLING, RECOMMENDED тАФ item G)

**Source:** the Bexar migration backlog тАФ
`runs/bexar_tx/backlog/v5.1.2-beta-framework-patches.md` item 1.

**Discrepancy note.** The Phase-5 instruction for this section described the
county-agnostic regression scanner as flagging "58 violations" caused by
`_VALID_US_STATE_CODES` and by county-side test fixtures (`BCAD-XXX`, `SAN ANTONIO/TX`
literals). The actual Bexar migration backlog does **not** document that framing. What it
*does* document (item 1, evidence-grounded) is below; the instruction's `_VALID_US_STATE_
CODES` / test-fixture framing could not be confirmed against the backlog and is recorded
at the end of this section as an **unverified candidate** for the cleanup session to
check, not asserted as fact.

**Problem (evidence-confirmed).** `scaffold/tests/test_county_agnostic_regression.py`
flags county-specific *pipeline output files* under `data/` тАФ e.g.
`data/leads_synthetic.json`, `data/runs/*.manifest.json`, `data/source_heartbeat.json` тАФ
as "county-agnostic leaks". These files are county-specific *by construction*:
`scaffold/data/README.md` documents `data/leads_synthetic.json` / `data/leads.json` as
the framework's canonical pipeline-output paths, and they carry the build's `county`,
`state`, and per-county lead records. Flagging them is a category error. After Phase 1
the scanner reported **20 violations**: 18 from `data/` output files and 2 from
`.claude/settings.local.json` (the operator-side bootstrap allow-list, which contains the
county name). Pipeline modules under `scaffold/pipeline/` and `scaffold/tests/` were
themselves clean.

**Proposed fix (per the migration backlog).** In the scanner's `is_exempt_path()`:

- exempt any path under `data/` тАФ these are county-specific build outputs documented in
  `scaffold/data/README.md`;
- also exempt `.claude/` тАФ operator-side Claude Code tooling configuration, not framework
  content;
- add a unit test asserting a synthetic `data/<slug>.json` does not fail the regression;
- document the exemption rules in `MASTER_PROMPT.md` alongside ┬з4.31.

**Unverified candidate (cleanup session to check).** The Phase-5 instruction additionally
suggested exempting references to a `_VALID_US_STATE_CODES` frozenset (universal data,
not county data) and files under `scaffold/tests/*` (test fixtures may legitimately carry
concrete `BCAD-XXX` / `SAN ANTONIO` literals). These are plausible exemptions and worth
evaluating, but they are not in the current Bexar backlog evidence тАФ the cleanup session
should confirm whether the scanner actually flags those sources before adding the
exemptions. The general principle holds either way: **the county-agnostic regression
scanner may flag violations from legitimate universal / output / test sources, and the
exemption list should cover them.**

This item is RECOMMENDED тАФ the scanner's false positives have a documented manual
workaround (scope the regression to framework directories explicitly), so it is not
blocking, but the exemptions remove recurring noise on every county build.
