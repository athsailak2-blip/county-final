# v5.1.2-beta framework patch backlog

This file logs framework-level patches surfaced during the Bexar build
that should ship in a future framework version. They are NOT applied in
this build (per operator directive 2026-05-14). Each item is captured
with: surface, evidence, proposed fix, scope.

## 1. test_county_agnostic_regression.py — exempt `data/`

**Surface (where the gap shows):** `scaffold/tests/test_county_agnostic_regression.py`
flags every county-specific output file under `data/` (e.g.
`data/leads_synthetic.json`, `data/runs/*.manifest.json`,
`data/source_heartbeat.json`) for containing the county name and
state string. The test currently exempts `config/counties/<slug>.json`
and `runs/<slug>/` but NOT `data/`.

**Evidence:** `scaffold/data/README.md` documents `data/leads_synthetic.json`
and `data/leads.json` as the framework's canonical pipeline output
paths. These outputs are county-specific by construction — they carry
the build's `county`, `state`, `build_label`, and the per-county lead
records. Marking them as "county-agnostic-leaks" is a category error.

After Phase 1 ran, the regression test reports 20 violations:
- 2 in `.claude/settings.local.json` (operator-side, pre-existing)
- 18 in `data/leads_synthetic.json` + `data/runs/*.manifest.json`
  (county-specific pipeline output, framework-documented)

Pipeline modules under `scaffold/pipeline/` and `scaffold/tests/` are
themselves clean (no county-specific strings).

**Proposed fix:** In `is_exempt_path()`:

```python
# County-specific build outputs. The data/ directory hosts
# pipeline-produced artifacts (leads.json, source_heartbeat.json,
# run manifests, evidence.jsonl). These are documented in
# scaffold/data/README.md as the framework's canonical output paths
# and are county-specific by construction.
if rel_str.startswith("data/"):
    return True
```

Consider ALSO exempting `.claude/` (operator-side Claude Code settings).
Currently `.claude/settings.local.json` contains the bootstrap-command
allow-list entry which includes the literal county name, triggering
two pre-existing violations on every Bexar build. Exempting `.claude/`
is reasonable because the directory is operator-tooling configuration,
not framework content.

**Scope:** v5.1.2-beta framework patch. Single-file change to
`scaffold/tests/test_county_agnostic_regression.py`. Add a unit test
that asserts a synthetic `data/<slug>.json` doesn't fail the
regression. Backwards-compatible.

**Workaround until patched:** Operators can manually verify their
Phase 1 build by running `scaffold/tests/verify_synthetic_harness.py`
(which exits 0 on a healthy synthetic harness) and ignoring the
data/-side `test_county_agnostic_regression.py` failures specifically
when they originate from county-specific output files. The framework's
intent — that `knowledge_base/`, `MASTER_PROMPT.md`, `scaffold/pipeline/`,
`scaffold/scrapers/`, `scaffold/ops/`, `scaffold/tests/` (except
test_county_agnostic_regression itself), `config/counties/_schema.json`,
`config/counties/_template.json`, and `README.md` stay county-agnostic
— is still verifiable by running the regression scoped to those
directories explicitly.

## 2. canonical_doc_types.json missing owner-name-pattern signal class

**Surface (where the gap shows):**
`knowledge_base/domain/canonical_doc_types.json` and
`knowledge_base/domain/02_signals_and_sources.md` enumerate signals as
they appear on recorded instruments and court filings only. There is no
declared signal class for "patterns inferred from parcel-master owner
strings" — even though such patterns can be a primary distress signal
in their own right.

**Evidence:** The Bexar operator established (2026-05-14, REVIEW_GATE_4
follow-up) that the following owner-name regex patterns, when present
on a parcel-master row, fire as signals on any lead joined to that
parcel:

- `ESTATE_OWNER_NAME_PATTERN`
  - Regex (case-insensitive): `\b(ESTATE OF|EST OF|ESTATE|HEIRS OF|HEIRS)\b`
  - Canonical signal: `estate` pattern
  - Confidence: 75 (medium — name string, not court-filed)
  - Operator rationale: owners whose parcel is held in an estate /
    heirship configuration are heir-hunting opportunities. Faster
    and broader than waiting for the probate court filing to surface
    via the court_civil source — the appraisal district sees the
    estate string the moment the owner of record changes.

- `LIVING_TRUST_OWNER_NAME_PATTERN`
  - Regex (case-insensitive): `\b(LIVING TRUST|FAMILY TRUST|REVOCABLE TRUST|REV TRUST|TRUST|TRUSTEE)\b`
  - Canonical signal: `transfer` pattern (living-trust ownership is a
    transfer instrument event)
  - Confidence: 70
  - Operator rationale: trust-owned property is a likely-deceased-or-
    incapacitated-owner heuristic; investor leverage when paired with
    distress.

- `ENTITY_OWNER_NAME_PATTERN`
  - Regex (case-insensitive): `\b(LLC|INC|CORP|LP|LTD|COMPANY|HOLDINGS)\b`
  - Canonical attribute: `entity_owned` (already exists in framework)
  - Confidence: 95
  - This is implemented as an ATTRIBUTE rather than a signal because
    the framework already recognizes entity_owned. Tightens the
    existing `_ENTITY_SUFFIXES` regex in `scaffold/pipeline/normalize.py`
    to the operator's authoritative set (drops TRUST/TR/TRUSTEE which
    fire LIVING_TRUST_OWNER_NAME_PATTERN instead).

**Proposed framework patch (v5.1.2-beta):**

Add two new entries to `canonical_doc_types.json`:

```json
"ESTATE_OWNER_NAME_PATTERN": {
  "lead_pattern": "estate",
  "subtype": "estate_owner_name_pattern",
  "source_class": "lead_generating",
  "default_confidence": 75,
  "document_priority": 55,
  "common_abbreviations": [],
  "notes": "Synthetic signal emitted by the pipeline when a parcel-master owner string matches an estate/heirship pattern (e.g. 'ESTATE OF', 'HEIRS OF'). Medium confidence: name match, not court-filed.",
  "source_priority": "P2"
},
"LIVING_TRUST_OWNER_NAME_PATTERN": {
  "lead_pattern": "transfer",
  "subtype": "living_trust_owner_name_pattern",
  "source_class": "lead_generating",
  "default_confidence": 70,
  "document_priority": 30,
  "common_abbreviations": [],
  "notes": "Synthetic signal emitted by the pipeline when a parcel-master owner string matches a living-trust ownership pattern (e.g. 'LIVING TRUST', 'REVOCABLE TRUST'). Trust ownership often signals deceased/incapacitated original owner.",
  "source_priority": "P2"
}
```

Add a new section to `02_signals_and_sources.md` covering
"Parcel-master derived signals" so the source-category checklist
recognizes owner-name patterns alongside recorded instruments.

**Bexar implementation (this build, not framework patch):**
Captured in `scaffold/pipeline/owner_name_patterns.py` (county-agnostic
module that emits the signals) and the synthetic CANONICAL.setdefault
blocks in `scaffold/pipeline/normalize.py`. The Bexar build runs
correctly without the framework patch; the patch is for cross-county
durability so other counties can reuse this signal class without
re-implementing it.

**Scope:** v5.1.2-beta. Two-line `canonical_doc_types.json` additions +
~20 lines of `02_signals_and_sources.md` prose. Add a unit test that
exercises both patterns on synthetic parcel rows. No breaking changes.
