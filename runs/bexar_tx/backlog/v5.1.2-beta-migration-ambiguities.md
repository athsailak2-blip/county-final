## Step 2 — Playbook validator command references nonexistent function

Date: 2026-05-15T15:48:41Z
Step: 2 (Move accepted_municipalities from code to config)

Ambiguity:
The playbook (line 135) and the operator's Step 2 brief both specify this
verification command:

    python3 -c "from scaffold.ops.write_county_config import validate; \
                print(validate('config/counties/bexar_tx.json'))"

But `scaffold/ops/write_county_config.py` does NOT export a `validate`
function. Its public API (`__all__`) is `["WriteResult", "write_county_config"]`.
`write_county_config()` is a write-and-validate-in-one-call API that takes a
Python dict and a target path; it has no read-only validate-existing-file
mode. Running the playbook command produces:

    ImportError: cannot import name 'validate' from
    'scaffold.ops.write_county_config'

This is a playbook-vs-code mismatch in canonical v5.1.2-beta, not something
introduced by migration work. The operator's Step 2 brief said:
"If validator complains, STOP and report." It is complaining (ImportError),
so I am stopping.

Options considered:
1. Use `write_county_config(cfg, path, schema_path=..., overwrite=True)`
   to round-trip the file — this both writes and validates against schema,
   and reports a structured WriteResult. Closest in spirit to playbook intent.
2. Import `jsonschema` directly and call `jsonschema.validate(cfg, schema)`
   — minimal, bypasses write_county_config entirely.
3. Use `jsonschema` CLI (`jsonschema -i config/counties/bexar_tx.json
   config/counties/_schema.json`).
4. Add a `validate(path)` helper to `scaffold/ops/write_county_config.py`
   that re-uses the internal schema-check code. Closest to making the
   playbook command "true" — but this is a framework-code change which
   the migration playbook does NOT authorize (changes to scaffold/ops/
   are not in Step 1's copy list and would violate "execute ONLY the 12
   steps").

What I would do absent operator input:
Option 1 — round-trip via `write_county_config(... overwrite=True)`. It
exercises the same schema-validation code path the playbook's `validate()`
presumably meant to invoke, returns a structured WriteResult, and does not
require any framework-code change. The downside is that it writes the file
back to disk (with the same content), which is mildly redundant but
harmless and atomic. I would NOT add a new `validate()` helper to the
framework module during the migration — that is a v5.1.2-beta canonical
fix that belongs upstream, not in a county-repo migration.

## Step 5 — On-disk data shape vs canonical translator input shape

Date: 2026-05-15T15:55:00Z
Step: 5 (Refactor build_leads.py for config-driven dispatch)

Ambiguity:
The v5.1.2-beta canonical translators in `scaffold/pipeline/translators/`
expect raw records in a specific shape:

  - A `raw_payload` dict wrapper on each record.
  - Inside `raw_payload`, the source's RAW field-names (for ArcGIS, the
    UPPERCASE attribute keys returned by the FeatureServer).
  - A `_layer_id` key inside raw_payload (with leading underscore).

(Verified by reading `scaffold/pipeline/translators/arcgis_foreclosure_notices.py`
docstring lines 38-55 and code line 152: `payload.get("_layer_id", "0")`;
`scaffold/pipeline/translators/arcgis_parcel_master.py` code line 107:
`payload = raw.get("raw_payload", {}) or {}`.)

Bexar's actual on-disk data in `data/raw/*.jsonl` is in a DIFFERENT shape:

`data/raw/foreclosure_notices_map.jsonl` — first record:
```json
{
  "raw_record_id": "...",
  "source_id": "...",
  "source_url": "...",
  "raw_payload": {
    "doc_type": "MORTGAGE",            // lowercase, not "TYPE"
    "doc_number": "20260600092",       // lowercase, not "DOC_NUMBER"
    "address": "3795 MOUNT OLIVE RD",  // lowercase, not "ADDRESS"
    "city": "ADKINS",                  // lowercase, not "CITY"
    "zip": "78101",                    // lowercase, not "ZIP"
    "recording_year": 2026,            // not "YEAR"
    "recording_month": 6,              // not "MONTH"
    "layer_id": 0,                     // not "_layer_id"
    ...
  }
}
```

`data/raw/parcel_master.jsonl` — first record (FLAT, no raw_payload wrapper):
```json
{
  "parcel_id": "BCAD-01372579",        // already prefixed + zero-padded
  "bcad_prop_id": 1372579,
  "situs_address": "10230 AMARYLLIS WAY",
  "situs_city": "BOERNE",
  "owner_name": "WILLIAMS SELDON VAUGHN & JASMIN",
  "year_built": 2022,
  "land_value": 82360.0,
  "improvement_value": 416680.0,
  "assessed_value": 499040.0,
  "exemptions": "DVHS, HS",
  "exempt_homestead": true,
  ...
}
```

Two distinct mismatches:

1. **Foreclosure**: `raw_payload` wrapper is present, but inner keys are
   LOWERCASE + use Bexar-scraper-specific names like `recording_year`
   (the canonical translator wants `YEAR`) and `layer_id` (the canonical
   translator hardcodes `_layer_id`). My Step-4 translator_config used
   UPPERCASE per the scraper's request side, but the on-disk data is the
   scraper's OUTPUT side, which is lowercase.

2. **Parcel master**: NO `raw_payload` wrapper at all. The data is in a
   pre-translated flat shape with framework-canonical field names already
   applied (`parcel_id`, `situs_address`, `owner_name`). The canonical
   translator can't even find the raw_payload to read from — it will
   silently translate zero parcels.

Root cause:
Bexar's county-specific scrapers in `scrapers/foreclosure_notices_map.py`
and `scrapers/parcel_master.py` were written PRE-v5.1.2-beta. They do
their own translation from raw ArcGIS attrs into normalized lowercase
form (foreclosure) or fully flat framework-canonical shape (parcel_master)
before persisting. The v5.1.2-beta canonical translators assume the
scrapers ONLY wrap raw ArcGIS output and the translation step is the
translator's job — so the translator expects raw UPPERCASE attrs.

Bexar's scrapers are NOT in the migration playbook scope (Steps 1-12 cover
`scaffold/`, `config/`, `knowledge_base/`, `dashboard/`, but not the
county-specific `scrapers/` directory).

Consequence:
If I plug the canonical translators into build_leads.py per playbook Step 5
without resolving this mismatch, the foreclosure translator will find none
of its configured field names in raw_payload (its `if not address or not
doc_number: continue` guard at line 154 will skip every record), and the
parcel_master translator will return zero parcels (because raw_payload
is empty for every flat record). Pipeline output: 0 leads, baseline: 287.
Step 12 regression check will trip the divergence protocol.

Options considered:

A) **Update translator_config in bexar_tx.json to match on-disk lowercase
   keys (foreclosure) and identity-map (parcel_master)**, AND add an
   in-orchestrator wrapper to add raw_payload to flat parcel_master records
   AND rename `layer_id` -> `_layer_id` for foreclosure. This contradicts
   what the operator approved at Step 4 ("Field-name verification at Step 4
   was rigorous — appreciated" referred to UPPERCASE = raw-ArcGIS names).
   Plus the translator hardcodes `_layer_id` (not configurable), so even
   with lowercase config the layer_id field needs an underscore rename
   at the orchestrator layer.

B) **Add a data-shape adapter in build_leads.py**: when reading
   `data/raw/<source>.jsonl`, transform each record so it conforms to the
   canonical raw-ArcGIS shape — for foreclosure, uppercase the inner keys
   and rename `layer_id` → `_layer_id`; for parcel_master, wrap the flat
   record into `{"raw_payload": <record>, "raw_record_id": ...}` and
   rename canonical fields BACK to BCAD's UPPERCASE schema (`parcel_id` →
   `PropID`, `situs_address` → `Situs`, etc.). Keeps Step 4's
   translator_config as approved (UPPERCASE raw-ArcGIS names) but adds a
   Bexar-specific transformer in the orchestrator — which is exactly what
   Step 8 says to scrub OUT of the pipeline.

C) **Rewrite Bexar scrapers** to produce raw ArcGIS shape (UPPERCASE keys
   in raw_payload, `_layer_id`, no flat fields). Then re-scrape. This is
   far outside Step 5 scope. It also throws away the existing 287-lead
   regression baseline because the underlying raw data changes.

D) **Hybrid**: keep the canonical translator approach as the long-term
   architecture, but write a one-time Bexar shim that lives in
   `scrapers/` (county-specific) and produces canonical-shape records
   from the existing data/raw/ files on the fly. The orchestrator loads
   data through this shim. Build_leads.py stays county-agnostic.

E) **Stop and ask**: the on-disk data shape is the de-facto baseline; the
   v5.1.2-beta canonical translator architecture assumes a different shape.
   This is a non-trivial architectural question — does v5.1.2-beta intend
   scrapers to be passthrough? If so, Bexar's scrapers need migration too
   (out of playbook scope). If not, translators need a "post-scraper shape"
   option that the playbook doesn't currently specify.

What I would do absent operator input:
Option B with a clean narrow scope: write a `_canonicalize_raw_record()`
helper in build_leads.py that's clearly labeled "TEMPORARY SHIM for
pre-v5.1.2-beta Bexar scraper output. Remove when scrapers are migrated."
The shim takes one record + the source_id, returns a record in canonical
raw-ArcGIS shape. Step 4's translator_config stays as UPPERCASE-approved.
The shim is acknowledged-county-specific and noted as a follow-up.

But this is a meaningful deviation from the playbook's clean
"all_signals.extend(translator_output)" pattern. It also leaves Step 8's
"scrub county-specific code from pipeline" with a literal shim to either
move into scrapers/ or scrub later.

Operator decision needed BEFORE Step 5 can produce a working build:
1. Confirm Option B (orchestrator shim) is acceptable, OR
2. Direct me to update translator_config to match on-disk keys (Option A,
   which requires re-validating Step 4's config — non-destructive,

3. Direct me to add scrapers/ rewrite to migration scope (Option C), OR
4. Direct me to write a Bexar-side scraper shim that produces canonical
